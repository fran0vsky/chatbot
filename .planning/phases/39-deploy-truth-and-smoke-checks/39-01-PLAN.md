<objective>
Add a cheap, unauthenticated, side-effect-free readiness endpoint at `GET /api/health` that the CI
post-deploy smoke stage (plan 39-02) can probe against the live domain to confirm the deploy is healthy
AND that `web_search` is actually configured on the running container. Today there is no way to tell — from
the outside — whether `TAVILY_API_KEY` reached the container; the `web_search` tool just silently returns
"Search unavailable" at call time (apps/backend/src/app/agents/tools/web-search.tool.ts). This endpoint
turns that invisible config gap into a green/red CI signal.

Purpose: a black-box health probe that reports tool-configuration readiness without leaking secrets.
Output: a `HealthController` at `/api/health`, registered in `AppModule`, with a unit test.
</objective>

<design_decisions>
- **Route (D-01):** `@Controller('health')` with a single `@Get()` → `/api/health`. Plain JSON, HTTP 200.
- **Shape (D-02):** Return `{ status: 'ok', tools: { web_search: boolean } }`. `status` is a constant 'ok'
  string (process liveness — if the process answers, it is up); `tools.web_search` is the readiness boolean.
  Keep the shape small and stable so the smoke `jq`/grep assertion in 39-02 is trivial.
- **Readiness derivation (D-03):** `web_search` = `TAVILY_API_KEY` is a non-empty string:
  `const key = process.env['TAVILY_API_KEY']; const webSearch = typeof key === 'string' && key.length > 0;`
  This mirrors exactly what `web-search.tool.ts` treats as "configured".
- **No secret leakage (D-04):** Only the boolean is emitted — never the key value, never its length. The
  unit test asserts the serialized response does not contain the stubbed key value.
- **Cheap + safe (D-05):** No DB query, no model call, no auth. Synchronous handler. CI hits it on every
  deploy and it costs nothing.
- **Optional shared type:** Inlining the return type in the controller is acceptable; a shared
  `@org/shared-types` interface is NOT required for an internal ops endpoint. Do not over-engineer.
- **Out of scope:** Reporting DB connectivity, OpenRouter reachability, or other tools' readiness — keep this
  plan to the single signal the phase needs (web_search). Additional readiness flags can be added later
  without changing the contract.
</design_decisions>
