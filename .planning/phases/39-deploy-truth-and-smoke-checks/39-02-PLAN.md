<design_decisions>
- **Job placement (D-01):** New job `smoke`, `needs: deploy-backend`, `if: github.ref == 'refs/heads/main'`,
  `runs-on: ubuntu-latest`. It runs only after the backend container is live. No GCP auth needed — it hits the
  public HTTPS domain with plain `curl`.
- **Domain source (D-02):** Use the live URL `https://dinoagents.duckdns.org`. Prefer a repo variable if one
  exists/added (e.g. `${{ vars.PUBLIC_URL }}`) with a literal fallback to keep it working today; literal is
  acceptable since the domain is already hardcoded in ROADMAP/README. Pick one and keep it readable.
- **Check 1 — dinos (D-03):** `curl -fsS -o /dev/null -w '%{http_code}' $URL/api/dinos` must be `200`; assert
  the body is a non-empty JSON array (`curl -fsS .../api/dinos | jq -e 'type == "array" and length > 0'`).
- **Check 2 — streamed chat (D-04):** `POST $URL/api/agents/chat` with `Content-Type: application/json` and
  body `{"message":"Reply with exactly the word PONG.","dinoId":"rexford"}`. Stream with
  `curl -N --max-time 60` and assert the captured output contains BOTH a `"type":"token"` line and a
  `"type":"done"` line. A short tool-free prompt avoids web_search calls and keeps cost/time minimal.
- **Retry for free-model 429 (D-05):** Wrap Check 2 in a bounded retry (up to 3 attempts, short sleep
  between) because free OpenRouter models 429 transiently; only fail the job if all attempts fail or no
  `done` event arrives within the timeout. Checks 1 and 3 are deterministic and need no retry.
- **Check 3 — web_search config (D-06):** `curl -fsS $URL/api/health | jq -e '.tools.web_search == true'`.
  This asserts the Phase 38 Tavily wiring reached the live container. (If Phase 38's Tavily secret is not yet
  deployed when 39 first runs, this check correctly fails — that is the intended signal, not a false alarm.)
- **Fail-fast (D-07):** Use `set -euo pipefail` in the run step so any failed assertion exits non-zero and
  fails the pipeline. Echo a clear ✓/✗ line per check for log readability.
- **Remove deploy-frontend (D-08):** Delete the entire `deploy-frontend` job. Do NOT touch `build-backend-image`,
  `deploy-backend`, `lint-test`, or `e2e`.
- **Keep storybook working (D-09):** `deploy-storybook` currently has `needs: deploy-frontend`. Repoint to
  `needs: e2e` so removing deploy-frontend doesn't orphan it. Storybook deploy is out of this phase's scope —
  leave its `continue-on-error: true` and steps untouched, only fix the `needs` reference.
- **Out of scope:** Deleting the GCS bucket itself / `FRONTEND_BUCKET` variable (infra teardown, not CI), and
  repurposing the bucket. PROD-05's "removed or repurposed" is satisfied by removing the job.
</design_decisions>
