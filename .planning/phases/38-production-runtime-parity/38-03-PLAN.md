<objective>
Automate database schema migrations so a change to `schema.ts` reaches the production Cloud SQL database
on deploy — eliminating the manual `drizzle-kit push` / manual `ALTER TABLE` workflow that silently
drifted prod behind the schema (the `when_to_activate` column was missing in prod and broke skill-save
until it was added by hand on 2026-06-11). The fix runs drizzle's programmatic `migrate()` at container
boot using committed SQL migrations.

Why boot-time (not CI): the runtime image excludes `drizzle-kit` (a devDependency), and Cloud SQL's
public IP is reachable only from the VM's authorized network — a GitHub Actions runner cannot connect.
The VM container is already authorized and already has `drizzle-orm` (a prod dependency, which ships the
node-postgres migrator). Boot-time `migrate()` is therefore the only place migrations can run with prod
dependencies against an authorized client.

Purpose: schema changes ship safely and automatically; no human ever runs a manual push for prod again.
Output: a shared connection helper, a `migrate.ts` runner called from `main.ts`, the Dockerfile copying
`drizzle/`, and a committed baseline migration crafted to apply cleanly to the existing prod database.
</objective>

<design_decisions>
- **Boot-time migrate(), not CI push (D-01):** Use `migrate` from `drizzle-orm/node-postgres/migrator`,
  invoked once at startup before `app.listen`. Rationale in <objective>: devDep absence + Cloud SQL
  authorized-networks make CI-side migration infeasible.
- **Shared connection helper (D-02):** Extract the connection-string normalization + ssl resolution from
  `database.module.ts` into `apps/backend/src/app/database/connection.ts` exporting
  `buildPoolConfig(url: string): PoolConfig`. Refactor `database.module.ts` to call it (no behavior
  change — the same Pool options result). `migrate.ts` uses the same helper so the migrator connects
  identically to the runtime pool (same SSL behavior on Cloud SQL public IP).
- **Guarded + degrades (D-03):** `runMigrations()` returns early (logs "DATABASE_URL not set — skipping
  migrations") when the URL is absent, matching the module's null-db path so local/e2e runs are
  unaffected.
- **Migrations folder resolution (D-04):** Resolve the folder as `process.env['MIGRATIONS_DIR']` else
  `join(__dirname, '..', 'drizzle')` — at runtime `__dirname` is `/app/dist`, so `/app/drizzle`. The
  Dockerfile copies the built `drizzle/` there (see Task 5). Locally, `drizzle-kit`/`migrate` resolve via
  the repo path; document the local invocation.
- **Baseline that fits an existing DB (D-05 — the key decision):** Generate the initial migration with
  `drizzle-kit generate`, then HAND-EDIT that single `0000_*.sql` so every `CREATE TABLE` →
  `CREATE TABLE IF NOT EXISTS` and every `CREATE INDEX` → `CREATE INDEX IF NOT EXISTS`. Because the file
  is edited BEFORE it is ever applied, the migrator hashes the edited content at apply time — so on the
  existing prod DB the baseline creates nothing, succeeds, and is recorded as applied; on a fresh/dev DB
  it creates everything. This avoids ANY manual prod SQL or journal surgery. Future migrations produced
  by `drizzle-kit generate` (ALTER TABLE, etc.) are committed as-is and need no editing.
  - Alternative considered + rejected: manually inserting a `__drizzle_migrations` baseline row in prod
    (operationally riskier, prod-specific, doesn't help dev/fresh). The IF-NOT-EXISTS baseline is the
    recommended path; if the operator prefers the journal-row approach, it can be swapped at execution —
    flag it in the SUMMARY.
- **Failure behavior (D-06):** On a migration error, log at `error` level with an unmistakable banner and
  re-throw so bootstrap aborts BEFORE listening — a backend serving against a half-migrated schema is
  worse than a visible failed deploy (the previous container keeps running because the new one never
  becomes healthy; `--restart=always` will retry, surfacing the problem). Document this fail-fast choice;
  if the operator wants log-and-continue instead, it's a one-line change noted in the SUMMARY.
- **No schema change in this plan:** This plan establishes the pipeline; `schema.ts` is unchanged. The
  baseline migration simply codifies the current schema. The pipeline is proven by Task 6 (add a throwaway
  column on a dev DB, regenerate, confirm auto-apply, then revert).
- **Out of scope:** switching prod to the Cloud SQL Auth Proxy, seed data, and down/rollback migrations
  (drizzle migrations are forward-only; rollback is a new forward migration).
</design_decisions>

<threat_model>
| Threat ID | Category | Component | Disposition | Mitigation |
|-----------|----------|-----------|-------------|------------|
| T-38-03-01 | Data integrity | Baseline on existing prod DB | mitigate | `IF NOT EXISTS` baseline is a no-op on existing tables and is recorded as applied, so no destructive DDL runs against live data; future migrations are reviewed SQL committed to the repo. |
| T-38-03-02 | Availability | Failed migration at boot | mitigate | Fail-fast (re-throw, abort before listen) means a bad migration prevents the new container becoming healthy; the previous container keeps serving and the failure is visible in deploy logs, rather than serving a half-migrated schema. |
| T-38-03-03 | Information Disclosure | DATABASE_URL in migrator | accept | The migrator reuses `buildPoolConfig` and the same secret-sourced URL as the runtime pool; no new exposure beyond the existing DB connection. |
| T-38-03-04 | Concurrency | Two containers migrating at once | mitigate | drizzle's migrator takes a transaction-scoped advisory lock during migration; the deploy stops the old container before starting the new one (vm-deploy.sh), so concurrent migration is not expected. |
</threat_model>
