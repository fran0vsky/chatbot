<objective>
Make `web_search` work on the production website by wiring `TAVILY_API_KEY` through the deploy path.
Today the tool reads `process.env['TAVILY_API_KEY']` (Phase 31), but `scripts/vm-deploy.sh` only
injects `OPENROUTER_API_KEY`, `DATABASE_URL`, and `CORS_ORIGIN` — the Tavily key is never passed and no
`tavily-api-key` secret exists in Secret Manager, so every search on the live site returns
"Search unavailable: TAVILY_API_KEY is not configured." This plan closes that gap and audits the full
env contract so no other backend-read variable is silently missing in production.

Purpose: a search-capable dino returns real results on https://dinoagents.duckdns.org, and the
localhost↔website env contract is single-sourced and complete.
Output: a Tavily fetch+inject in `vm-deploy.sh`, a verified-complete `.env.example`, and an
INFRASTRUCTURE.md note documenting the new secret + a one-time human task to create it.
</objective>

<design_decisions>
- **Secret name (D-01):** `tavily-api-key` — kebab-case, consistent with `openrouter-api-key` and
  `database-url`.
- **Graceful degradation (D-02):** `TAVILY_KEY=$(fetch_secret tavily-api-key 2>/dev/null || echo "")`.
  If the secret doesn't exist yet, the deploy still succeeds and `web_search` degrades to its existing
  "unconfigured" string — exactly today's behavior, no regression. The key turns search on the moment
  the secret is created (next deploy), with no code change.
- **Injection (D-03):** Add `-e TAVILY_API_KEY="$TAVILY_KEY"` to the `docker run` block alongside the
  existing `-e` flags. Keep ordering/readability consistent with the others.
- **Secret creation is a human task (D-04):** Creating the Secret Manager secret with the real key value
  is a one-time `gcloud secrets create` operation requiring the operator's Tavily key and gcloud auth —
  it cannot and must not be done from this repo. Document the exact command in INFRASTRUCTURE.md and as a
  manual task here; the script change is safe to ship before the secret exists (D-02).
- **Env-contract audit (D-05):** Cross-check the grep'd `process.env[...]` reads against (a) `.env.example`
  and (b) `vm-deploy.sh` injections. Expected outcome: all of OPENROUTER/DATABASE_URL/CORS_ORIGIN/TAVILY
  injected; PORT + NODE_ENV come from the Dockerfile (`ENV PORT=3000`, `ENV NODE_ENV=production`) so they
  need not be injected; BODY_LIMIT is optional (has a code default) but SHOULD appear in `.env.example`
  (handled by 38-01). Record any genuine gap found; if none, state the contract is complete.
- **Out of scope:** rotating the existing secrets, Cloud SQL Auth Proxy, and the CI workflow itself
  (vm-deploy.sh is the injection point; CI already passes the script over SSH unchanged).
</design_decisions>
