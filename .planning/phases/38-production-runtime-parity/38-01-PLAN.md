<objective>
Raise the backend's HTTP request-body size limit so chat requests carrying an attached image
(the frontend downscales to a ~1024px JPEG data URL — typically 150–500 KB of base64, well over
Express's silent 100 kb default) plus a capped conversation history are accepted instead of being
rejected with HTTP 413. This gap affects BOTH localhost and the website; it is the reason Phase 25/26
image-attach UAT was never able to pass end-to-end through the API.

Purpose: image-attach and long-history chats work; the cap is explicit, documented, and env-tunable.
Output: a body-parser limit override in `main.ts` (json + urlencoded), and a `BODY_LIMIT` entry in
`.env.example`.
</objective>

@.planning/ROADMAP.md
Entry point: [apps/backend/src/main.ts](../../../apps/backend/src/main.ts) — bootstraps a
`NestExpressApplication`, calls `app.enableCors(...)`, `app.setGlobalPrefix('api')`, conditionally
`app.useStaticAssets(...)`, then `app.listen(port)`. There is currently NO body-parser configuration,
so Express's built-in default of 100 kb applies to every JSON request.
Where the large bodies come from: the chat composer attaches images as base64 data URLs. The
downscaler in [libs/ui/src/lib/input-composer/input-composer.ts](../../../libs/ui/src/lib/input-composer/input-composer.ts)
caps the longest edge at 1024px and JPEG-encodes at quality 0.85, then the data URL is sent as
`imageDataUrl` in the chat request body (see
[apps/backend/src/app/agents/agents.controller.ts](../../../apps/backend/src/app/agents/agents.controller.ts)).
History (capped at 20 turns) rides in the same body.
Backend rules: [apps/backend/CLAUDE.md](../../../apps/backend/CLAUDE.md) — read env via
`process.env['VAR']`, no `any`, document every new env var, use NestJS `Logger` not console.
NestExpressApplication exposes `useBodyParser(type, options)` (Nest 11) — the idiomatic way to raise
the limit without manually importing express middleware. Either that or `app.use(express.json({ limit }))`
BEFORE `setGlobalPrefix` is acceptable; `useBodyParser` is preferred because the app is already typed
as `NestExpressApplication`.
Note: Caddy (prod reverse proxy) does not impose its own small body cap by default, and the original
Phase 36 nginx config that set `client_max_body_size 25m` is NOT in the live Caddy path — so the
backend limit is the real ceiling in production.
</context>

<design_decisions>
- **Mechanism (D-01):** Use `app.useBodyParser('json', { limit })` and
  `app.useBodyParser('urlencoded', { limit, extended: true })` on the `NestExpressApplication`. This is
  the Nest-idiomatic override and keeps the existing parsers (no double-registration). Apply it before
  `app.listen`; placement relative to `enableCors`/`setGlobalPrefix` does not matter for body parsing.
- **Default + env (D-02):** `const bodyLimit = process.env['BODY_LIMIT'] || '10mb';`. 10 MB comfortably
  covers a 1024px JPEG data URL (≤ ~1.5 MB base64) plus a 20-turn history with up to 2 retained images
  (the Phase 32 image cap), with generous headroom, while still bounding abuse. The value is a
  body-parser size string (e.g. `'10mb'`).
- **Both parsers (D-03):** Apply the limit to json AND urlencoded so no body shape silently falls back
  to 100 kb.
- **No new logging noise:** Do not log on every request. Optionally log the resolved limit once at
  bootstrap via the existing `Logger` if it reads cleanly, but this is discretionary.
- **Out of scope:** Caddy/proxy body caps (the live Caddy default is large enough; revisit only if a
  413 appears at the proxy layer in Phase 44 UAT), and multipart/file-upload parsing (images arrive as
  base64 JSON, not multipart).
</design_decisions>
