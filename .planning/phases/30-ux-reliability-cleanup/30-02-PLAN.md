<objective>
Fix the composer textarea so large text never breaks the pill layout. Vertical growth is already capped at 8
rows with internal scroll; the remaining defect is horizontal overflow from a `flex-1` textarea without `min-w-0`.

Purpose: the composer stays intact no matter what the user pastes — a baseline reliability fix.
Output: a constrained textarea that caps height, scrolls, and stays inside the pill.
</objective>

`autoResize()` ([input-composer.ts:202](../../../libs/ui/src/lib/input-composer/input-composer.ts#L202)) already caps height at
`lineHeight * 8` and sets `atMaxHeight` to toggle `overflow-y-auto`. The textarea is `flex-1` inside a
`flex items-center gap-1` row ([input-composer.html:153](../../../libs/ui/src/lib/input-composer/input-composer.html#L153)) but
has no `min-w-0`; flex items default to `min-width:auto`, so a long unbroken token forces the textarea (and the pill)
wider than its container. `autoResize` runs on `(input)` but not when `draft` is set programmatically (e.g. `usePrompt`,
STT transcript), which can leave a tall textarea unrendered until the next keystroke.
</context>

<design_decisions>
- Primary fix: add `min-w-0` to the textarea so the flex item can shrink and wrap instead of overflowing horizontally.
- Confirm vertical behavior already holds (8-row cap + `overflow-y-auto` via `atMaxHeight`). Keep as-is unless repro
  shows otherwise.
- Trigger `autoResize` after programmatic `draft` changes so height is correct without requiring a keystroke. The
  composer is OnPush; call autoResize after the view updates (e.g. in a `ngAfterViewChecked` guarded by a dirty flag,
  or expose a small method the host can't reach — simplest: re-run autoResize in `submit()` already happens; add a
  setter/effect for draft). Keep the change minimal and within the component.
</design_decisions>
