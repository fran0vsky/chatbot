<objective>
Add a groupchat mode: the user picks several dinos, sends one prompt, and each dino answers in its own attributed, independently-streaming panel. Reuses the existing single-dino streaming endpoint by fanning out one request per dino — no backend change required.

Purpose: Compare perspectives at a glance; showcases the cast of dinos.
Output: A GroupchatService (parallel streams), a presentational GroupResponse panel, and a groupchat view in the chat shell.
</objective>

ChatService.streamMessage already streams a single dino via SSE (dinoId-aware after Phase 18/19). The chat shell uses an `activeView` signal. Each groupchat stream is just another streamMessage call with a distinct dinoId and its own threadId.
</context>

<design_decisions>
- No backend endpoint needed: the frontend opens N independent SSE streams (one per selected dino) via the existing ChatService. Each gets its own threadId (`group-{groupId}-{dinoId}`) and AbortController.
- Add `activeView` value `'groupchat'`. The composer in this view sends to all selected dinos.
- GroupResponse is presentational (one per dino). GroupchatService (smart) owns the parallel state (signals keyed by dinoId: streamingText, status, error).
- Cap selectable dinos (e.g. 2–4) to bound free-model load.
- Within-thread history per dino panel is optional in v1 (each panel can be single-turn or keep its own history array) — keep v1 single-turn per send to stay simple; note multi-turn group history as a future enhancement.
</design_decisions>
