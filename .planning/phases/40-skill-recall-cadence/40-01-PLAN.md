<objective>
Replace the current "inject every taught skill on every turn" recall with the mentor's cadence:
**once per conversation, pull the single most relevant taught skill.** A cheap LLM scorer matches the
conversation's opening user message against each skill's `whenToActivate` and returns the one best skill
(or none). Selection is computed against the opening message â€” recoverable from `history` â€” so it stays
stable across the whole thread with no DB persistence. The system prompt then carries at most one skill.
Which skill was chosen is logged and emitted as a new `skill_active` stream event for the UI (Plan 02).

Output: a `StreamSkillActiveEvent` in shared-types, a `selectRelevantSkill` method, a single-skill
`buildSystemPrompt`, and the wiring + log + event emission in `streamAgent`.
</objective>

<design_decisions>
- **Opening message (D-01):** `const openingMessage = history?.find(h => h.role === 'user')?.text ?? message;`
  Scoring against this guarantees the same skill is chosen on every turn of the thread without persistence.
- **Scorer (D-02):** New private `async selectRelevantSkill(openingMessage: string, skills: SkillView[],
  signal?: AbortSignal): Promise<SkillView | null>`. Returns `null` immediately when `skills.length === 0`.
  Otherwise asks `MEMORY_EXTRACTION_MODEL` to return the id (or 1-based index) of the single most relevant
  skill given the opening message and each skill's `whenToActivate`/`title`, or `NONE`. Parse the reply,
  map back to a `SkillView`; if the reply is `NONE`/unparseable/no match, return `null`. Whole body in
  try/catch returning `null` on error â€” mirrors `extractAndStoreMemories`. Bound it with a timeout (reuse
  the existing turn `signal`, or a short internal timeout) so a slow free model can't stall the turn.
- **Injection (D-03):** Change `buildSystemPrompt` to accept a single `skill: SkillView | null` (or keep the
  array param but pass a 0/1-length array â€” implementer's choice; single param is cleaner). When present,
  emit ONE standing-instruction block naming that skill (drop the "apply ALL of them in every single
  response" plural framing; one skill, applied this conversation). When null, omit the block entirely.
- **Emission + log (D-04):** In `streamAgent`, after selection: if a skill was chosen, `this.logger.log`
  its id+title and `yield { type: 'skill_active', skillId, skillTitle }`; else log "no skill selected".
  Emit before the token stream starts so the UI can show the hint early.
- **New event (D-05):** Add `StreamSkillActiveEvent { type: 'skill_active'; skillId: string; skillTitle:
  string }` and add it to the `StreamEvent` union. The controller's `write(event)` bridge already forwards
  any `StreamEvent` to SSE â€” no controller change needed.
- **Out of scope:** memories injection (unchanged), getSkills query (unchanged â€” still fetches all; we
  select among them in-process), any DB/schema change.
</design_decisions>
