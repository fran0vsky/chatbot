<objective>
Make group conversations **durable**. Today the groupchat is ephemeral; this plan saves a completed
turn-based group thread as a single interleaved `ConversationSession` in the same localStorage store as
normal chats (D-08), carrying the attributed message list (who said what, with pinned emoji reactions)
and the participant roster. The history panel lists group threads alongside single chats — visually
distinguished — and reopening one restores the full transcript top-to-bottom and the exact dino
selection, switching back into the groupchat view.

Purpose: deliver Success Criterion #4 / GRP2-04 — group conversations are saved in the history panel
and reopen with the full transcript — with no DB change and no migration, reusing the existing
store-driven session persistence path.
Output: extended `ChatMessage`/`ConversationSession` types, `toSession()`/`loadSession()` on
GroupchatService with persistence on completion, the reopen branch in ChatComponent, a group-thread
indicator in HistoryPanel, and persistence/reopen tests.
</objective>

<design_decisions>
- **Type extensions (D-08).** `ChatMessage` += optional `dinoId?: string` (speaker for a group dino
  message) and `reactions?: GroupReaction[]` (pinned emoji chips). `ConversationSession` += optional
  `isGroup?: boolean` and `participantDinoIds?: string[]` (the saved roster). All optional → existing
  single-chat sessions remain valid (no migration).
- **Transcript mapping.** `GroupMessage ⇄ ChatMessage`:
  - to ChatMessage: `role 'user'`→`'user'`; `role 'dino'`→`'assistant'` with `dinoId`; carry `text`,
    `reactions`, `createdAt`.
  - from ChatMessage: `'assistant'` + `dinoId` → group `'dino'` message; `'user'` → group `'user'`;
    preserve id (reuse `createdAt`/a stable id), `reactions`.
- **Persist on completion (D-08).** GroupchatService gains `toSession(title): ConversationSession`
  (`isGroup: true`, `participantDinoIds`, mapped messages, a stable `groupSessionId`). When a group turn
  finishes (`group_done`) AND the transcript has ≥1 user message, ChatComponent persists it via the same
  store path single chat uses (dispatch the session upsert / `HistoryService.upsertSession`) then
  dispatches `SessionActions.loadSessions()` so the panel refreshes. Title = first user message (seeded
  like single chat). The group session keeps a stable id across turns in the same session so re-saving
  updates rather than duplicates.
- **Reopen (Success Criterion #4).** Add `openSession(session)` in ChatComponent. The history-panel
  `sessionSelected` handler branches: if `session.isGroup` → `setActiveView('groupchat')`,
  `groupchatService.loadSession(session)` (restores the `messages` signal from the mapped transcript and
  returns the roster), set `selectedGroupDinoIds(session.participantDinoIds ?? [])`, close the sidebar;
  else keep the existing `setActiveView('chats'); switchToSession(session)` path. Single-chat reopen is
  unchanged.
- **Panel indicator.** HistoryPanel renders a group affordance when `session.isGroup` — a small cluster
  of participant mascots (from `participantDinoIds`, via `app-mascot`) or a group icon — so group threads
  are distinguishable at a glance. Update the story with a group-session example.
- **No DB, no migration (D-08).** Everything rides the existing localStorage `ConversationSession` store;
  rejected DB tables / per-dino sub-threads per CONTEXT.md.
</design_decisions>
