<objective>
Replace the parallel fan-out UI with the **turn-based group chat client**. Rebuild `GroupchatService`
around the single backend endpoint `POST /api/agents/group` (Plan 01): consume the orchestrator `plan`
to lay out Round-1 answerers in order, route concurrent dino token streams into their slots, pin emoji
**reaction chips** to their target messages, and append the bounded Round-2 replies — so the thread
reads top-to-bottom like a real chat with clear per-dino attribution. Add `@mention` autocomplete to
the group composer (forces a reply) and send the accumulated attributed transcript as `history` so
multi-turn group conversations stay coherent. The old per-dino fan-out (`DinoStreamEntry[]`,
`group-{groupId}-{dinoId}` threads) is removed with no fallback.

Purpose: deliver the user-facing half of Success Criteria #1 (per-dino replies/reactions — GRP2-01),
#2 (`@mention` forcing + volunteered dissent rendered with attribution — GRP2-02), and #3 (interleaved
top-to-bottom transcript with inter-dino replies — GRP2-03). Persistence (#4) is Plan 03.
Output: a `streamGroup()` on ChatService, a rebuilt GroupchatService + spec, an extended group-response
row (reaction chips + respondingTo), and the rewired groupchat view with `@mention` autocomplete.
</objective>

<design_decisions>
- **streamGroup on ChatService.** Add `async *streamGroup(message, participantDinoIds, history, signal):
  AsyncGenerator<GroupStreamEvent>` that POSTs a `GroupChatRequest` ({ message, participantDinoIds,
  userId: this.userId, history }) to `${environment.apiUrl}/api/agents/group` and reuses the exact
  SSE-frame parse loop from `streamMessage` (split on `\n\n`, strip `data: `, `JSON.parse` → yield).
  `streamMessage` is left exactly as-is.
- **GroupchatService rebuild.** Replace `DinoStreamEntry[] entries` with `messages = signal<GroupMessage[]>([])`
  and a `streaming = signal<boolean>(false)`. `send(message, participantDinoIds)`:
  1. abort any prior stream (`stopAll`), set `streaming(true)`;
  2. push the user message (role `'user'`, fresh id) onto `messages`;
  3. build `history` = the prior `messages` (excluding the just-added turn) capped at `HISTORY_CAP = 20`;
  4. consume `chatService.streamGroup(message, cappedIds, history, signal)`:
     - `plan`: for each `round1` `answer` decision (in `order`), append an empty placeholder dino message
       (role `'dino'`, `dinoId`, `status: 'streaming'`, fresh id) so Round-1 slots render in plan order;
     - `dino_token`: append `text` to that dino's current streaming message (match by dinoId + streaming);
     - `dino_done`: finalize that message (set full `response`, `status: 'done'`, store the server
       `messageId` so reactions can target it);
     - `reaction`: attach a `GroupReaction { dinoId, emoji }` to the message whose id === `targetMessageId`;
     - `dino_error`: mark that dino's message `status: 'error'` with the message;
     - Round-2 `dino_token`/`dino_done` for a dino with no open placeholder → create+append a new dino
       message (Round-2 replies append after Round-1 in arrival order);
     - `group_done`: `streaming(false)`.
  - Cap `participantDinoIds` to `MAX_DINOS` (keep the static `MAX_DINOS = 4`).
  - `stopAll()` aborts the single `AbortController` and clears `streaming`.
  - **Removed:** `DinoStreamEntry`, per-dino `controllers` map, `streamForDino`, `group-{groupId}-{dinoId}`
    threading. No fallback to the old fan-out.
- **group-response row (D-06).** Add `@Input() reactions?: GroupReaction[]` and `@Input() respondingToName?: string`.
  Render reaction chips (each = `app-mascot` of the reacting dino + emoji) pinned at the bottom of the
  message; when `respondingToName` is set, show a subtle "↳ replying to {name}" affordance in the header.
  The panel stays presentational (no services). Update the Storybook story with a reactions example.
  (`GroupReaction` comes from `@org/shared-types`; resolving a reacting dinoId → mascot uses `app-mascot
  [dinoId]`, so the chip needs only the dinoId + emoji.)
- **Interleaved rendering (D-03).** `chat.html` groupchat view replaces the response **grid** with a
  single top-to-bottom column: `@for (msg of groupchatMessages(); track msg.id)`. A `'user'` message
  renders as a user bubble (reuse the existing message bubble styling/`app-message-bubble` where it fits,
  or a simple Tailwind bubble); a `'dino'` message renders `app-group-response` with `[dino]`/`[text]`/
  `[status]`/`[error]`/`[reactions]`/`[respondingToName]`. The dino selector grid + composer stay.
- **@mention autocomplete (GRP2-02 / D-04) — app layer.** Implement in the groupchat view, not by
  rewriting `app-input-composer`. Track the composer draft; when the draft has a trailing `@<partial>`
  token, show a dropdown of participant dinos whose name matches `<partial>` (case-insensitive). Selecting
  one replaces the trailing token with `@Name ` (styled token) in the composer draft (set via the existing
  `inputComposerRef.draft` channel used elsewhere in chat.ts). On send, the `@Name` text travels in the
  message and the backend (Plan 01) does the forcing — the frontend only needs to insert the token.
- **chat.ts wiring.** `onGroupSend(text)` → `groupchatService.send(text, selectedGroupDinoIds())`.
  Expose `groupchatMessages = groupchatService.messages`. Add the mention-autocomplete state
  (`mentionQuery`, `mentionCandidates`, `applyMention`). Remove `groupchatEntries`/`groupDinoById` usage
  tied to the old fan-out (keep `groupDinoById` only if still needed to resolve a message's dino summary
  for `app-group-response`). Keep `toggleGroupDino`, `selectedGroupDinoIds`, `groupchatMaxDinos`.
- **No regression.** `streamMessage`, single chat, and Arena are not touched.
</design_decisions>
