<objective>
Build the backend **turn-based group orchestrator** that replaces the parallel fan-out. A single
cheap orchestrator call (`openai/gpt-4o-mini`) reads the user message + participant roster + recent
attributed transcript and returns a structured plan deciding, per dino, whether to **answer**, **react**
with one emoji, or stay **silent**, plus the speaking order and any inter-dino mentions. Only the
"answer" dinos then make full in-character `streamAgent` calls on their own registry models; "react"
and "silent" dinos cost nothing. Round 1 answerers stream concurrently (multiplexed, dino-tagged, onto
one SSE stream); a single bounded Round 2 lets up to a couple of dinos reply to what was just said,
run sequentially so each sees the Round-1 answers. `@mentions` force a reply.

Purpose: deliver the engine behind Success Criteria #1 (per-dino answer/react/silent — GRP2-01), #2
(forced mention + volunteered dissent — GRP2-02), and #3 (bounded inter-dino dialogue + attributed
transcript — GRP2-03), with a hard, documented per-turn LLM-call ceiling that carries forward the
Phase 23 `MAX_DINOS = 4` cap.
Output: shared group contracts, a new `GroupAgentsService` + `GroupAgentsController` wired into
`AgentsModule`, and unit tests for plan parsing, @mention forcing, the Round-2 cap, and the
react=no-LLM-call invariant.
</objective>

<design_decisions>
- **Architecture = backend-orchestrated single SSE (D-01).** One `POST /api/agents/group` call performs
  the orchestrator call + Round 1 + Round 2 and multiplexes every dino's events over one SSE stream.
  This gives the central control D-01 requires (enforce order, loop bounds, cost cap) rather than
  splitting orchestration across client round-trips.
- **Cost ceiling (D-01/D-02).** Constants in the service: `ORCHESTRATOR_MODEL = 'openai/gpt-4o-mini'`,
  `MAX_GROUP_DINOS = 4` (forward of Phase 23 `MAX_DINOS`), `MAX_INTER_DINO_REPLIES = 2`. Participants are
  capped to `MAX_GROUP_DINOS` before the orchestrator call. Round 2 is clamped to `MAX_INTER_DINO_REPLIES`.
  Worst-case LLM calls per user message = `1 + 4 + 2 = 7`, documented in a header comment.
- **Plan schema (D-07).** `DinoTurnDecision { dinoId; action: 'answer'|'react'|'silent'; emoji?; targetMessageId?; respondingTo?; order }`.
  `GroupOrchestratorPlan { round1: DinoTurnDecision[]; round2: DinoTurnDecision[] }`. A `react` decision
  carries exactly one `emoji`; `answer` carries neither emoji nor implies a reaction (one action per dino
  per round). `parseOrchestratorPlan(raw)` strips ``` fences, `JSON.parse`, coerces/validates: drops
  unknown dinoIds, forces `action` into the union (default `silent`), keeps only the first emoji char(s)
  of `emoji`, and truncates `round2` to `MAX_INTER_DINO_REPLIES`.
- **@mention forcing (GRP2-02 / D-04).** `parseMentions(message, roster)` derives forced dinoIds by
  matching `@<name>` tokens (case-insensitive, against participant dino `name`). After parsing the plan,
  any forced dino whose `round1` action ≠ `'answer'` is overridden to `'answer'` and appended to the
  Round-1 order if missing. The forced set is also named in the orchestrator prompt so ordering stays
  coherent.
- **Volunteered dissent (D-05).** The orchestrator system prompt instructs: a non-addressed dino should
  volunteer in Round 2 ONLY when it has a genuinely different take, a disagreement, or a concrete useful
  addition, and must set `respondingTo` to the dino it answers; otherwise stay silent (no echoing).
- **Attributed history (D-09).** `buildAttributedHistory(transcript, answeringDinoId, roster)` converts
  the group `GroupMessage[]` into `ChatHistoryItem[]` for a specific answerer: the answerer's own prior
  dino messages → `{ role: 'assistant', text }`; every other message (the user and other dinos) →
  `{ role: 'user', text: '<SpeakerLabel>: <text>' }` where the label is `User` or the dino's name;
  reactions are appended as a short note (e.g. `(Glyphos reacted 👍)`). The list is sliced to the most
  recent `HISTORY_CAP = 20` turns. This is what each `streamAgent` answerer receives as its `history`.
- **Round 1 concurrency, ordered rendering (D-03).** Start all `answer` dinos' `streamAgent` generators
  concurrently; merge their yielded events onto the SSE stream, re-tagging each as a `GroupStreamEvent`
  carrying `dinoId`. The first event emitted is `{ type: 'plan', plan }` so the frontend can pre-lay-out
  ordered slots and route subsequent `dino_token`/`dino_done` by `dinoId`. (Concurrency lives on the
  backend; the frontend renders strictly in `plan.round1` order.)
- **Round 2 sequential (D-02/D-03).** After Round 1 completes, for each `round2` decision in order:
  append the prior answers to the transcript, then run that dino's `streamAgent` with the updated
  attributed history so it genuinely sees what it reacts to; emit its dino-tagged events; reactions are
  emitted as `{ type: 'reaction', dinoId, emoji, targetMessageId }` with no LLM call.
- **Reactions cost nothing (D-01/D-07).** A `react` decision in either round emits only a `reaction`
  event built from the plan — `react`/`silent` dinos never call a model.
- **GroupStreamEvent union:** `plan`, `dino_token {dinoId,text}`, `dino_done {dinoId,response,messageId}`,
  `reaction {dinoId,emoji,targetMessageId}`, `dino_error {dinoId,message}`, `group_done`. Each answer
  carries a server-generated `messageId` so reactions can target it.
- **Graceful failure.** An answerer error becomes a `dino_error` event for that dino (the others still
  complete). The orchestrator call itself wraps in try/catch: on failure, fall back to a minimal plan =
  every participant `answer` in roster order, Round 2 empty (so the group still responds). The stream
  always ends with `group_done`.
</design_decisions>
