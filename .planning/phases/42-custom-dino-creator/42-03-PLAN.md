built-in OR a DB-loaded custom dino mapped to the `Dino` shape, threaded through the single-chat loop and the
group engine, plus a merged `GET /api/dinos?userId=` so custom dinos appear in the picker. This is the
resolution + merge layer — no avatar upload (Plan 02) and no UI (Plan 04). It also closes the four code-review
findings against Plan 01's shipped service. After this plan, selecting a custom dino actually chats with its
authored prompt + gated tools, and a custom dino can join a group chat (CDINO-02, CDINO-04).
</objective>

<design_decisions>
- **Resolver (D-01):** New `dino-resolver.ts` exports `async resolveDino(id: string | undefined, userId: string | undefined, customDinoService: CustomDinoService): Promise<Dino | undefined>`. If `id` starts with `custom:` → `customDinoService.getById(id, userId)`, map to the `Dino` shape via `customDinoToDino(custom)`; if not found → `undefined` (NOT a silent fallback to the default built-in — a missing custom dino must not impersonate Rexford). Else → `getDino(id)`. `customDinoToDino` is exported and unit-tested.
- **Mapping (D-02):** `customDinoToDino` produces `{ id, name, species: custom.species ?? '', persona: custom.persona ?? '', blurb: custom.blurb ?? '', specialty: 'Custom dino', model, systemPrompt, toolNames, accent, avatarUrl }`. `imageGen` is omitted/false (custom dinos are not image-gen in this phase). The mapped `toolNames` are exactly the persisted (already catalogue-validated) names; `resolveActiveTools` still intersects them at chat time.
- **Single chat (D-03):** Inject `CustomDinoService` into `AgentsService`. Replace `const dino = dinoId ? getDino(dinoId) : undefined;` with `const dino = await resolveDino(dinoId, userId, this.customDinoService);`. Behaviour for built-ins and the no-dino case is unchanged. (`userId` is already a `streamAgent` param.)
- **Group chat (D-04):** Inject `CustomDinoService` into `GroupAgentsService`. Make `resolveRoster` async (`await resolveDino` per id, drop ids that resolve to `undefined`, dedupe, cap to `MAX_GROUP_DINOS`) and thread `userId` into it. `isImageGenDino` is consulted with an already-resolved roster dino's `imageGen` flag rather than calling `getDino(id)` again — refactor the call site in `streamGroup` to read `dino.imageGen` from the resolved roster `Dino` it already holds (avoids a second lookup and works for custom dinos). `streamGroup` receives `userId` already.
- **Merged endpoint (D-05):** `DinosController` injects `CustomDinoService`. `list(@Query('userId') userId?: string)` returns `DINOS.map(toDinoSummary)` plus, when `userId` is present, `(await customDinoService.list(userId)).map(...)` projected through the existing `toCustomDinoSummary`. Keep `systemPrompt` out (allowlist already guarantees this).
- **DinoSummary.avatarUrl (D-06):** Add `avatarUrl?: string` to `Dino` in shared-types; `DinoSummary = Omit<Dino,'systemPrompt'>` inherits it. `toDinoSummary` (dinos.ts) adds `avatarUrl: dino.avatarUrl` to its allowlist (built-ins leave it undefined). The custom projection already carries `avatarUrl`.
- **Review fixes folded in (D-07):**
  - **CR-02:** At the top of `update()`, after the prefix guard, reject when no field is present: `if (!Object.keys(req).some(k => (req as Record<string,unknown>)[k] !== undefined)) throw new BadRequestException('update request must include at least one field');`
  - **WR-01:** In the `create()` and `update()` `catch (err)` blocks, `if (err instanceof HttpException) throw err;` before logging + returning null, so validation 400s surface.
  - **WR-03:** Add a `validateAvatarUrl(url)` private that requires an `http:`/`https:` absolute URL; call it in `validate()` (create) and in `update()` when `avatarUrl !== undefined`.
  - **WR-04:** `customDinos.updatedAt` gains `.$onUpdate(() => new Date())`; the manual `updatedAt: new Date()` may stay (harmless) or be removed.
- **Scope fence:** Do NOT touch `memory-creator.service.resolveDino` here — its `getById`-style narrowing is for the teach flow and is out of the four roadmap truths; leaving it on built-in resolution does not block any SC. (Noted as a known follow-up, not a gap.) No frontend files. No avatar upload.
</design_decisions>
