<design_decisions>
- **Table (D-01):** `customDinos` columns per 42-CONTEXT specifics. `toolNames` is `jsonb('tool_names').notNull().default([])`; index on `userId`. Export `CustomDino`/`NewCustomDino` inferred types.
- **Id namespace (D-02):** Persist the uuid plainly in the `id` column, but the service maps DB rows to a public id of `` `custom:${row.id}` `` and strips the prefix on lookups. (Keeps the FK-free uuid clean while giving the resolver a cheap prefix branch.) Document this mapping in one place in the service.
- **Validation (D-03):** On create/update the service rejects (`BadRequestException`) a `model` not in `MODEL_CATALOGUE` and any `toolNames` entry not in the tool catalogue. Trim/require `name` and `systemPrompt`.
- **Catalogue (D-04):** `MODEL_CATALOGUE` is a typed array of `{ id, label }` seeded from the free/cheap registry models. `isAllowedModel(id)` is the single guard used by the service and reused in Plan 03's resolver.
- **Projection (D-05):** The service exposes a `toCustomDinoSummary(row)` that yields a `DinoSummary`-compatible object (id with `custom:` prefix, name, species, persona/blurb, model, toolNames, accent, plus `avatarUrl`) and NEVER includes `systemPrompt`. Plan 03's merged `/api/dinos` reuses it.
- **No avatar upload here:** `avatarUrl` is just a string column/field; the upload endpoint that fills it is Plan 02. Create/update accept an optional `avatarUrl` string.
- **Env:** add `AVATAR_BUCKET` to `.env.example` now (documented; consumed in Plan 02) so the env contract is single-sourced.
</design_decisions>
