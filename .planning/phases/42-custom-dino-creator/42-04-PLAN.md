<design_decisions>
- **Custom detection (D-01):** A summary is a custom dino when `id.startsWith('custom:')`. The card and picker use this to show the avatar + edit/delete affordances; built-ins are unchanged.
- **DinoService userId (D-02):** Add `private readonly userId = loadUserId()` (import from chat.service.ts). `fetchDinos()` and `loadDinos()` call `GET /api/dinos?userId=<id>` so the merged roster includes custom dinos. CRUD methods send `userId` (in body for create; query param for PUT/DELETE, matching the live controller).
- **DinoService methods (D-03):** Return Observables (consistent with `fetchDinos`):
  - `fetchModels(): Observable<CuratedModel[]>` → `GET /api/models`
  - `uploadAvatar(file: File): Observable<{ url: string }>` → multipart `POST /api/custom-dinos/avatar` (FormData, field `file`)
  - `createCustomDino(req: Omit<CreateCustomDinoRequest,'userId'>): Observable<CustomDino>` → `POST` with `userId` merged in
  - `updateCustomDino(id, req: UpdateCustomDinoRequest): Observable<CustomDino>` → `PUT /api/custom-dinos/:id?userId=`
  - `deleteCustomDino(id): Observable<void>` → `DELETE /api/custom-dinos/:id?userId=`
- **CustomDinoCreator (D-04):** New smart standalone OnPush component in apps/frontend. Inputs: `@Input() editing?: DinoSummary` (edit mode when set). Outputs: `@Output() saved` and `@Output() cancelled`. Internal signals back the form fields (name, blurb, persona/reaction prompt = the form's "how it reacts" textarea mapped to `systemPrompt`, avatarUrl, model, toolNames[]). On open it `fetchModels()` to populate the dropdown; in edit mode it seeds fields from `editing` (note: `systemPrompt` is not in the summary, so edit pre-fills name/blurb/persona/model/tools/avatar and leaves the prompt empty with a hint that re-saving replaces it — acceptable for this phase). Avatar file input → `uploadAvatar` → sets `avatarUrl` (show a spinner/disabled save while uploading). Save calls create or update via DinoService, emits `saved` on success; errors set an inline `error` signal and never throw.
- **Form field mapping (D-05):** The personality/reaction prompt textarea maps to `systemPrompt` (the authored "how it reacts" prompt). `description` maps to `blurb`. `persona` (one-line tagline) is optional; if the form keeps a single prompt field, set `persona` to a short derived/empty value and put the full text in `systemPrompt`. Name + prompt + model are required to enable Save; tools default to none.
- **Picker/card (D-06):** `DinoCard` gains `@Output() editDino` + `@Output() deleteDino` and renders `<img [src]="dino.avatarUrl">` when `avatarUrl` is set, else the existing `<app-mascot>`. Edit/delete buttons render only when the card is custom (`dino.id` starts with `custom:`) — pass an `@Input() custom = false` computed by the picker to keep DinoCard logic-free. `DinoPicker` gains `@Output() addDino`, renders an "add a dino" tile/button alongside the grid, and re-emits `editDino`/`deleteDino` from its cards.
- **ChatComponent wiring (D-07):** Add a `dinoCreatorOpen` signal + `editingDino` signal. Handlers: `openDinoCreator()` (add — distinct name from the existing skill `openCreator()`), `onEditDino(d)`, `onDeleteDino(d)` (confirm via `window.confirm`, then `deleteCustomDino` → reload), `onDinoSaved()` (close creator + `store.dispatch(DinoActions.loadDinos())` + markForCheck). Wire `(addDino)`, `(editDino)`, `(deleteDino)` on both `<app-dino-picker>` instances; mount `<app-custom-dino-creator>` in an overlay mirroring the skill overlay (`@if (dinoCreatorOpen())`).
- **Roster refresh (D-08):** After create/update/delete, dispatch `DinoActions.loadDinos()` so the NgRx roster (and both pickers) refresh. No reducer change needed.
- **Scope fence:** No backend changes (Plan 02/03 own those). No NgRx schema change beyond re-dispatching `loadDinos`. Group-chat selection needs no new code — custom dinos arrive in the roster and the existing `toggleGroupDino` already works by id.
</design_decisions>
