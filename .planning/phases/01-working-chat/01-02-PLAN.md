<objective>
Build the Angular bubble chat UI on top of the existing Angular SPA skeleton: a three-level standalone component tree (App → app-chat → app-message-bubble), a dedicated injectable ChatService that owns the per-session UUID threadId and posts ChatRequest to the backend, full-viewport layout per D-13 through D-16, auto-expanding textarea with Enter-to-send (D-08 / D-09), typing-dots indicator (UI-03), auto-scroll to newest (UI-04), and inline red error bubbles for failed requests (D-03 through D-07).

Purpose: This is the user-visible half of the walking skeleton. After Plan 01 the backend returns real LLM answers; this plan makes the user able to send messages and see the answers in a chat layout. Combined with Plan 01 it satisfies all of UI-01, UI-02, UI-03, UI-04, CONV-01, and CONV-02 — and runs end-to-end locally against `npx nx serve backend` + `npx nx serve frontend`.

- New `apps/frontend/src/environments/{environment.ts, environment.prod.ts}` with `apiUrl` and a `fileReplacements` entry in `apps/frontend/project.json` so the prod build swaps in `environment.prod.ts`
- New `apps/frontend/src/app/chat/` directory containing `chat.types.ts`, `chat.service.ts`, the `chat.ts` host component, and a nested `message-bubble/` component
- Root `App` updated to import and render `<app-chat />` instead of `NxWelcome`; the `NxWelcome` file deleted
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user input → DOM | Untrusted text typed in the textarea is interpolated into Angular templates |
| browser → backend API | Frontend `ChatService` POSTs `ChatRequest` to the backend at `${environment.apiUrl}/api/agents/chat` |
| environment.prod.ts → bundle | The production `apiUrl` is baked into the static bundle deployed to Firebase Hosting |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | XSS via assistant response | mitigate | Bubble component renders `message.text` via Angular text interpolation `{{ message.text }}` — Angular escapes HTML by default. Markdown rendering is OUT of scope per ENH-02; no `[innerHTML]` is used anywhere in this plan. |
| T-02-02 | Tampering | XSS via user message echoed back | mitigate | Same as T-02-01 — user-typed text is treated as plain text in the template, never injected as HTML. |
| T-02-03 | Information Disclosure | Error bubble copy | mitigate | Per D-05, error bubble shows a generic "Something went wrong. Please try again." — no HTTP status, no stack trace, no backend error body surfaced. |
| T-02-04 | Information Disclosure | apiUrl in production bundle | accept | Cloud Run service URL is intentionally public (it's the API endpoint the frontend calls). No secrets baked into the bundle. `OPENROUTER_API_KEY` lives only in Cloud Run Secret Manager (Plan 03). |
| T-02-05 | Denial of Service | Repeated send while loading | mitigate | Both the textarea and the send button bind `[disabled]="isLoading"` (D-10), preventing double-send. `send()` also short-circuits if `isLoading` is true. |
| T-02-06 | Spoofing | CSRF against backend | accept | Backend has no auth and no cookies; CSRF is not applicable. CORS already restricts origins to `http://localhost:4200` in dev and the Firebase URL in prod (set in Plan 03 via `CORS_ORIGIN`). |
| T-02-07 | Repudiation | Per-tab thread isolation | mitigate | `ChatService` uses `providedIn: 'root'`, so each browser tab/page-load gets a fresh service instance and a fresh `crypto.randomUUID()` threadId (CONV-02). No cross-tab leakage. |
| T-02-SC | Tampering | npm supply chain | mitigate | This plan adds NO new dependencies — uses only `@angular/*`, `rxjs`, and `@org/shared-types`, all already in the lockfile. No fresh installs introduced. |
