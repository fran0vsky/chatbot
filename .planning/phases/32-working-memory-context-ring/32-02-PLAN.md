<objective>
Add a **context-usage ring** to the composer so the user can see, live, roughly how full the next turn's
context window is ‚ÄĒ with a warning as it nears the limit. The estimate must reflect the same material the
backend will assemble (Plan 32-01's replayed history text + retained images + reused tool results) plus the
current draft, measured against the active dino model's real context window (D-07), and warn at ~80%
without auto-removing anything (D-09/D-10).

Purpose: a context-usage ring is visible in the chat reflecting approximate context-window fill, with a
visible near-limit warning (CTX-03 / Success Criterion #3).
Output: a per-model window map + token-estimation helpers in `@org/shared-types`, a presentational
`UsageRing` donut in `@org/ui` (with a Storybook story), optional ring inputs on `InputComposer`, and a
live `contextUsage` estimate wired up in the chat component.
</objective>

<design_decisions>
- **Per-model window map (D-07):** Add `MODEL_CONTEXT_WINDOWS: Record<string, number>` in shared-types
  with the registry models above (use ~128000 each; these are approximate per D-08 and the implementer may
  refine), plus `gpt-4o-mini` (128000). `DEFAULT_CONTEXT_WINDOW = 8000` is the conservative fallback for
  any unknown model. `getContextWindow(modelId)` returns the mapped value or the default. Keep this in
  shared-types (not backend-only `model-capabilities.ts`) so the **client-side** ring can import it ‚ÄĒ this
  is the deviation from CONTEXT.md's "extend model-capabilities.ts" note, chosen because the ring must run
  live in the browser and `model-capabilities.ts` is backend-only / not importable from the frontend.
- **Token estimate (D-08, approximate & documented):**
  - `estimateTextTokens(text) = Math.ceil(text.length / 4)` (char/4 heuristic).
  - `IMAGE_TOKEN_COST = 1000` flat tokens per retained image (approximation; documented).
  - The chat component sums: every replayed history item's `text` (√∑4) + `toolResult` (√∑4) +
    `IMAGE_TOKEN_COST` per retained image (capped at the same 2 as 32-01) + a fixed
    `SYSTEM_PROMPT_ALLOWANCE` (‚Čą800 tokens, since the client doesn't hold the server system prompt) +
    the current draft (√∑4). All heuristics, explicitly approximate per the ROADMAP scope note.
  - Percent = `min(100, round(estimate / getContextWindow(model) * 100))`.
- **Live update:** Expose the estimate as a `computed()`/signal in `chat.ts` derived from `messages()`,
  the draft, and `activeDino().model`. The draft lives in the composer; surface it to the chat component
  via an existing or new `(draftChange)` output, OR compute the draft contribution inside the composer.
  Implementer's choice: simplest is the composer owns its `draft` and the chat passes the message/image
  baseline as inputs, with the composer adding its own draft tokens before rendering the ring. Keep the
  estimation *function* in shared-types so both options stay testable and free of domain logic.
- **Warn-only at 80% (D-09, D-10):** Single warning state at `percent >= 80` ‚ÄĒ ring stroke shifts to a
  warning colour (Tailwind amber/red token consistent with the jungle theme) and the tooltip notes it is
  near the limit. Nothing is trimmed or removed.
- **Visual (D-12):** A small SVG **donut** (two concentric circle strokes; foreground arc via
  `stroke-dasharray`) showing the % fill, with a `title`/tooltip of `~{tokens} tokens (~{percent}%)`.
  Tailwind classes only; no inline styles beyond the SVG geometry attributes (`r`, `cx`, dasharray) which
  are SVG presentation attributes, not CSS `style=`.
- **Placement (D-11):** Embed `<app-usage-ring>` inside `InputComposer`'s toolbar row, immediately before
  the send button, gated by an `@Input() showContextRing`/non-null percent so other composer usages
  (groupchat/arena) that don't pass it render nothing.
</design_decisions>
