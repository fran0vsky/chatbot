<objective>
Make earlier **attached images** and **fetched tool results** reusable on later turns within a chat
thread. Today `buildHistory()` maps each prior turn to `{ role, text }` only — it strips `imageDataUrl`
and silently drops the `role: 'tool'` messages (they have empty text), so the model loses access to an
image attached two turns ago and re-downloads a page `fetch_page` already pulled. This plan threads that
data through the existing client→backend `history` channel and reconstructs faithful multimodal
HumanMessages + AIMessage/ToolMessage pairs in the stateless agent loop.

Purpose: an image attached earlier stays referenceable without re-attaching (CTX-01 / Success Criterion #1),
and a fetched page is reused instead of re-downloaded on follow-ups (CTX-02 / Success Criterion #2), with
**zero** change to single-turn behavior (Success Criterion #4).
Output: an extended `ChatHistoryItem`, a richer `buildHistory()`, a reconstructing `historyMessages`
mapper, and backend unit tests.
</objective>

<design_decisions>
- **Contract extension (D-03, D-04):** Widen `ChatHistoryItem` to
  `role: 'user' | 'assistant' | 'tool'` and add optional `imageDataUrl?: string`,
  `toolName?: string`, `toolArgs?: Record<string, unknown>`, `toolResult?: string`. All new fields
  optional → existing `{ role, text }` items remain valid (no-regression guarantee at the type level).
- **Image cap N = 2 (D-01, D-02):** In `buildHistory()`, retain `imageDataUrl` on only the **last 2**
  image-bearing user turns; strip it from older ones (they go as text). Retained images are sent every
  turn (deterministic, no retrieval logic — D-02). N=2 is within the suggested 2–3 band and bounds
  vision-token cost.
- **Forward tool messages (D-04):** Stop filtering out `role: 'tool'` items. They carry empty `text`
  but real `toolName`/`toolArgs`/`toolResult`. Forward them as `ChatHistoryItem`s with role `'tool'`.
- **Backend replay (D-04, D-05):** In the `historyMessages` mapper, expand each item:
  - `user` with `imageDataUrl` → multimodal `HumanMessage` (same `{ content: [ {type:'text'...},
    {type:'image_url', image_url:{ url }} ] }` shape as the current turn).
  - `user` without image → `HumanMessage(text)`.
  - `assistant` → `AIMessage(text)`.
  - `tool` → **two** messages: `new AIMessage({ content: '', tool_calls: [{ id, name: toolName,
    args: toolArgs ?? {} }] })` then `new ToolMessage({ content: toolResult ?? '', tool_call_id: id })`,
    where `id` is a synthetic stable id per item (e.g. `replay-${toolName}-${index}`). This is a faithful
    replay of what the model already "has", so it won't needlessly re-call — but the tool stays in
    `activeTools`, so the model MAY re-fetch when it judges freshness matters (D-05). Use a `flatMap`
    since one item can yield two messages.
  - **Reconstruction is approximate-faithful** (CONTEXT.md flags it as the implementer's task): the exact
    `tool_calls` array key/shape must match what `@langchain/openrouter` accepts for an assistant
    tool-call message; mirror the structure the live loop already produces on `response.tool_calls`.
- **HISTORY_CAP:** Keep the 20-turn cap, but apply it to **user/assistant turns** and let the `tool`
  messages interleaved within the kept window pass through with them (a turn may be user→tool→tool→
  assistant). Do not let tool messages consume the turn budget. Implementer's discretion on the exact
  slice mechanic provided the kept window is the most-recent ~20 conversational turns plus their tools.
- **No tool-result cap here (D-06):** Reused tool text is already bounded by Phase 31 (`fetch_page` ~5KB,
  search top-5). Global growth is governed by the context ring (Plan 32-02) + HISTORY_CAP, not a new cap.
</design_decisions>
