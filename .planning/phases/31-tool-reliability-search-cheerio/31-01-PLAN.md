<objective>
Replace the DuckDuckGo Instant Answer `web_search` implementation with Tavily, a real free-tier
search provider built for LLM agents. The tool must return the top 5 real results (title + snippet +
source URL) for a query, and degrade to clear, distinct, model-readable strings when Tavily is down,
rate-limited, or the key is unconfigured.

Purpose: a current-events query returns relevant, usable results the model can cite and chain from —
not the sparse, mostly-empty DDG Instant Answers.
Output: a rewritten `web-search.tool.ts`, a new `web-search.tool.spec.ts`, and a `TAVILY_API_KEY`
entry in `.env.example`.
</objective>

<design_decisions>
- **Tavily call (D-01, D-02, D-12):** POST `https://api.tavily.com/search` with a JSON body of
  `{ query, search_depth: 'basic', topic: 'general', max_results: 5 }` and the key in the
  `Authorization: Bearer ${apiKey}` header (Tavily's current auth; passing `api_key` in the body is an
  accepted equivalent — implementer's choice, pick one). No `time_range`, no recency flag, no extra
  schema args. Keep a `User-Agent` header consistent with the existing tool.
- **Result shape (D-04, D-05):** Map `response.results` (each `{ title, url, content, score }`) to the
  top 5. Render each as `title` + a snippet of `content` capped at ~200 chars (ellipsis on overflow) +
  the `url`. All 5 pass through (~1–1.5KB total) — replace the old 600-char global `MAX_LEN`. Exact
  per-result formatting is implementer's discretion (e.g. numbered block with the URL on its own line).
- **Degradation (D-08):** Distinct returned strings, no fallback provider:
  - Key missing (`!apiKey`): return before any fetch, e.g. `Search unavailable: TAVILY_API_KEY is not configured.`
  - HTTP 429: e.g. `Search rate-limited: the search provider is over quota, try again later.`
  - Any other non-OK status or thrown/network error: e.g. `Search failed: <reason>` (include status or error message).
  - Empty `results` array on a 200: keep a `No results found for: ${query}` string.
  Wording is implementer's discretion provided the three cases are clearly distinguishable.
- **Types:** Define a small `interface TavilyResult { title?: string; url?: string; content?: string }`
  and `interface TavilyResponse { results?: TavilyResult[] }`. No `any` — narrow `await res.json()` as
  `unknown` then cast/validate to `TavilyResponse`, or type the parse directly. Remove all DDG interfaces.
- **Description (D-14):** Refresh to note it returns fresh web results WITH source URLs that can be
  passed to `fetch_page` for deeper reading. Keep `schema: z.object({ query: z.string().min(1).max(200) })`.
</design_decisions>
