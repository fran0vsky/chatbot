semantic heuristic finds nothing.

<design_decisions>
- **Dependency (D-06):** Add `cheerio` only — no `@mozilla/readability`, no `jsdom`. Install with
  `npm install cheerio --legacy-peer-deps` so it lands in the root `package.json`/`package-lock.json`
  (single-package workspace). Import as `import { load } from 'cheerio';`.
- **Extraction heuristic (D-06, D-07):** `load(html)`; remove `script, style, noscript, nav, footer,
  aside, header, form, svg` (and similar chrome). Pick the content root: first non-empty of `article`,
  `main`, `[role=main]`; else the element with the largest combined text length among candidate
  containers (e.g. `body` descendants). Walk the root in document order, emitting `h1..h6` headings and
  `p`/list/text blocks as separate lines so structure is preserved. Title from `$('title').first().text()`
  (or `<h1>` fallback). Collapse whitespace per block. Implementer owns the exact largest-block heuristic
  and heading/paragraph formatting.
- **Cap (D-07):** Keep `MAX_TEXT_LEN = 5000` on the final assembled text (ellipsis on overflow), same as today.
- **Soft-fallback (D-09):** If the assembled main-content text is empty/whitespace after extraction, fall
  back to cleaned whole-`body` text (`$('body').text()` collapsed) rather than returning empty. Only if
  THAT is also empty return a clear "could not extract readable content" string.
- **Size hardening (D-10):** After a successful response, before reading/parsing: if `Content-Length`
  header exists and exceeds a `MAX_BYTES` cap (~2–3MB, pick a concrete constant e.g. `2_500_000`), abort
  with a clear oversized error. Since `Content-Length` may be absent, also bound the read — read the body
  and if its byte/char length exceeds `MAX_BYTES`, return the oversized error before Cheerio parsing.
  Keep the existing 10s `AbortSignal.timeout`.
- **Content-type gating (D-11):** Read `content-type`.
  - Contains `html` → Cheerio extraction path.
  - Starts with `text/` (non-html) or contains `json` → cleaned raw text, capped at `MAX_TEXT_LEN` (no Cheerio).
  - Anything else (binary: `application/pdf`, `image/*`, `application/octet-stream`, etc.) → return a clear
    `Unsupported content type: ${contentType}` error WITHOUT reading/dumping the bytes.
- **Schema/description:** Keep `name: 'fetch_page'` and `schema: z.object({ url: z.string().url() })`.
  Description may be lightly refreshed to mention it extracts main content (title/headings/body); not required to change.
</design_decisions>
