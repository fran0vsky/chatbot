<objective>
Add a model selector to the chat UI so users can switch between GPT-4o mini and Claude 3 Haiku during a session.

Three concrete changes:
1. Update `ChatService.sendMessage()` to accept and forward a `model` parameter in the `ChatRequest` body.
2. Add `selectedModel` state to `ChatComponent` — a property or signal initialized to `'openai/gpt-4o-mini'` — and pass it when calling `sendMessage()`.
3. Add a native `<select>` element to the chat header (right side, per D-05) styled with Tailwind, bound to `selectedModel`, and disabled while `isLoading` is true (per D-07).

After this plan, all three Phase 2 success criteria are met: selector is visible, selection drives the model used, and selection persists for the session duration.
</objective>

<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user → Angular component | User selects a model from a pre-defined list in the UI — no free-text model ID entry |
| Angular component → backend | `selectedModel` is one of two hardcoded IDs; no user-constructed string reaches the API |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation |
|-----------|----------|-----------|-------------|------------|
| T-02-04 | Tampering | Model ID in ChatRequest | mitigate | The frontend `models` array contains only two IDs (`openai/gpt-4o-mini`, `anthropic/claude-3-haiku`); the `<select>` option values are bound to this array. A user cannot inject an arbitrary model ID through the UI. The backend also applies a fallback gate (Plan 02-01 T-02-01) for any tampered request. |
| T-02-05 | Information Disclosure | Angular build | accept | Model IDs are visible in the compiled JS bundle — this is acceptable; they are not secrets. |

</threat_model>
