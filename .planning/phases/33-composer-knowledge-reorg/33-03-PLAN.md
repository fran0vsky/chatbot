<objective>
Make taught skills first-class and editable everywhere they appear. Extend the presentational
`<app-skill-manager>` with inline (expand-in-row) editing of name / trigger / instruction and a
`skillEdited` output, add `SkillService.updateSkill`, and render the same edit-capable list in BOTH the
teach modal and the Knowledge view (which today only does file upload). One `SkillService` data source
feeds both surfaces. Create + delete behavior is unchanged (CMP-05).

Purpose: a user can view and edit a saved skill (not just delete it), from the teach modal or the
Knowledge view, with the change persisted through plan 01's `PUT /api/skills/:id`.
Output: an edit-capable `SkillManager` (+ story), `SkillService.updateSkill`, chat `onSkillEdited` wiring,
and a Knowledge-view skills section reusing the shared component.
</objective>

<design_decisions>
- **Inline expand edit (D-09):** Editing happens INSIDE the skill row in `<app-skill-manager>`, not by
  reusing the teach form. Internal component state: `editingId: string | null`, `editTitle`,
  `editWhenToActivate`, `editInstruction` plain fields. `startEdit(skill)` seeds them; the row swaps its
  static view for inputs (title), an optional "when to activate" input, and an instruction textarea, plus
  Save/Cancel. `cancelEdit()` clears `editingId`. `saveEdit()` emits `skillEdited` and clears edit state.
- **skillEdited output (D-10):** `@Output() skillEdited = new EventEmitter<{ id: string; title: string;
  whenToActivate?: string; instruction: string }>()`. Save trims; disable Save when title or instruction is
  empty.
- **Service (D-10):** `updateSkill(id, { title, whenToActivate, instruction })` → `PUT ${base}/skills/${id}`
  with body `{ title, whenToActivate, instruction }`, returns `Observable<DinoSkill>`. Also extend `addSkill`
  to optionally accept+send `whenToActivate` (keeps create-form trigger support from D-08, optional).
- **Shared reuse (D-11):** The Knowledge view (chat.html ~388, inside the `@else if (activeView() ===
  'knowledge')` block) gains a skills section rendering the SAME `<app-skill-manager>` bound to
  `learnedSkills()` with `(skillDeleted)` and `(skillEdited)` (memories may be passed `[]` there or the full
  list — pass `[]` to keep Knowledge focused on skills). Both surfaces are fed by `SkillService` via the
  chat host's `learnedSkills` signal.
- **Load on Knowledge view (CMP-04):** Ensure learned skills are loaded when the Knowledge view is shown
  (reuse the same load used by `openSkillPanel`), so the list is populated without opening the teach modal.
- **onSkillEdited:** Chat handler calls `skillService.updateSkill(...)`, and on success replaces the matching
  entry in `learnedSkills` (`update(s => s.map(x => x.id === updated.id ? updated : x))`).
</design_decisions>
