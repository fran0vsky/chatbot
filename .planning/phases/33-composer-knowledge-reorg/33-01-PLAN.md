<objective>
Add the data layer for first-class, editable skills. Introduce a nullable `whenToActivate` ("trigger")
field on skills end-to-end — shared type, Postgres column, and read/write paths — and expose a new
`PUT /api/skills/:id` endpoint plus a `MemoryService.updateSkill` method so a saved skill can be
updated (not just created/deleted). Existing always-apply behavior for skills without a trigger must
not change (CMP-05).

Purpose: the frontend edit UI (plan 02) and Phase 34's AI Memory Creator can read and write a skill's
name / trigger / instruction through a real API.
Output: updated `DinoSkill` type, a migrated `dino_skills` table, `updateSkill` service logic with the
column threaded through `getSkills`/`addSkill`, and the new PUT route.
</objective>

<design_decisions>
- **Field name & nullability (D-06/D-07/D-08):** `whenToActivate?: string` on `DinoSkill` and `SkillView`;
  DB column `when_to_activate text` — nullable, no `.notNull()`, no default, no backfill. Blank/whitespace
  input persists as SQL `NULL`.
- **Always-apply default (D-07, CMP-05):** Do NOT change how skills are gated. `buildSystemPrompt` keeps
  injecting every skill. The trigger is rendered only as an OPTIONAL hint appended to a skill's line when
  present, e.g. `- ${title}: ${instruction}` becomes `- ${title} (use when: ${whenToActivate}): ${instruction}`
  ONLY if `whenToActivate` is non-empty. Skills without a trigger render byte-identically to today.
- **Update method (D-10):** `updateSkill(id, { title, whenToActivate, instruction })` — trim title/instruction,
  coerce blank `whenToActivate` to `null`, `UPDATE ... WHERE id = id RETURNING {id,title,instruction,whenToActivate}`.
  Return the row or `null`. No (userId × dinoId) re-scoping on update — mirror the existing un-scoped
  `deleteSkill(id)` (anonymous-userId model; see threat model).
- **PUT route (D-10):** `@Put('skills/:id')` — require a non-empty `title` and `instruction` (400 otherwise);
  `whenToActivate` optional. `null` from the service → `ServiceUnavailableException` (DB off), matching POST.
- **getSkills/addSkill:** add `whenToActivate: dinoSkills.whenToActivate` to every select/returning projection
  so the field flows through GET and POST responses.
</design_decisions>
