<objective>
Give each dino a persistent, per-user memory that carries across threads. Add a `user_memories` table scoped by (userId × dinoId), a MemoryService that writes durable facts and retrieves them, and wire the agent loop to (a) inject relevant memories into the system prompt and (b) finally send within-thread conversation history so multi-turn works. Identity is an anonymous per-device id (no auth yet). All DB use degrades gracefully when `DATABASE_URL` is absent.

Purpose: "The dino learns you" — the differentiator from a stateless chatbot.
Output: New schema + MemoryModule/MemoryService + dino-loop integration + a userId plumbed from the client.
</objective>

<design_decisions>
- Identity: the frontend generates and persists an anonymous `userId` in localStorage and sends it on every chat request (`ChatRequest.userId`). `sessions.userId` is already nullable text — reuse the same id.
- Memory model (v1, simple + robust): store short natural-language fact strings per (userId, dinoId). Retrieval is "most recent N" (e.g. 20) injected into the prompt — NOT vector search (keep it free + simple; semantic retrieval is a future enhancement).
- Extraction (v1): after a turn completes, run a lightweight extraction — either a tiny LLM call asking "list any durable facts about the user worth remembering (or NONE)" OR a heuristic. Prefer a SMALL/cheap model call gated so it only runs when the turn likely contains a durable fact. Keep it best-effort; failures never break the chat.
- Within-thread history: the frontend already holds `messages[]`. Send the recent slice (cap ~20 messages / a token budget) so follow-ups have context. This closes the stateless-loop gap and is required for memory extraction to make sense.
- Graceful degradation: MemoryService checks `db` for null and no-ops (returns []/does nothing) when DB is disabled.
</design_decisions>
