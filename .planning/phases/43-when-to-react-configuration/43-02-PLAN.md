<design_decisions>
- **Component split (D-07, project component-architecture rule):** the **presentational** control lives in `libs/ui` (no injected service, `@Input`/`@Output` only, OnPush, standalone, Storybook story — mirrors `SkillManager`). The **smart** part (HTTP + state) is `ReactivityService` in `apps/frontend`, wired through the existing `ChatComponent`. No new smart container component is added — `ChatComponent` is the host, exactly as it hosts `SkillManager`.
- **Control widget:** a per-row segmented control over `REACTION_LEVELS` (4 buttons / radio group), label per level. Executor may use a select if cleaner; must be keyboard-operable and Tailwind-styled in the desert/jungle theme. Iterate over `REACTION_LEVELS` (no hardcoded level list in the template).
- **Roster source:** the panel lists the **currently selected group dinos** (the same `DinoSummary[]` the group picker uses), so it is a group-chat setting. Built-in and `custom:` dinos render identically (both are `DinoSummary`).
- **Default display (D-05/SC#4):** a dino missing from the levels map renders as `normal`. The panel never writes on open — only an explicit user change calls `setLevel`.
- **Optimistic update:** on `levelChanged`, update the `levels` signal immediately, then PUT; on PUT error, log + keep the optimistic value (degraded like the rest of the app — null-db backends just no-op). No blocking spinner.
- **Surfacing:** mount the panel behind an existing group-chat affordance (e.g. a settings toggle near the group dino picker), parallel to how `SkillManager` is revealed. Exact placement is executor discretion within `chat.html`.
</design_decisions>
