<objective>
Introduce NgRx as the frontend state foundation and expose a whitelisted catalogue of dispatchable app actions. Migrate the app's core state (active dino, theme, active session + messages, active view) from ad-hoc component fields/signals into typed store slices, and refactor ChatComponent to select + dispatch. This is the precondition for the voice "dino assistant" (Phase 29), which will fire catalogued actions.

Purpose: A single, inspectable source of truth and a safe, named action surface the assistant can drive.
Output: @ngrx/store + effects wired, dino/ui/session slices, an ACTION_CATALOGUE whitelist, and ChatComponent refactored with zero behavior change.

WARNING: This is a REFACTOR, not a feature. The bar is "identical behavior, better architecture." Migrate incrementally; guard against regressions at each step.
</objective>

<design_decisions>
- Classic @ngrx/store (actions/reducers/selectors) + @ngrx/effects — chosen over SignalStore specifically because NGX-02/Phase 29 need a named, enumerable action surface.
- Migrate the HIGH-VALUE, assistant-relevant slices first: `ui` (theme, activeView, panels), `dino` (roster, activeDinoId), `session` (sessions list, activeSessionId, messages). Keep transient streaming state as component signals for now (not assistant-relevant) to bound the refactor — document this boundary.
- Effects wrap existing services (HistoryService persistence, DinoService load, theme localStorage) so side-effects move out of the component without changing behavior.
- ACTION_CATALOGUE is the ONLY surface the assistant may dispatch through (Phase 29). It maps a stable intent name → { description, paramsSchema (zod), create(params) => Action }. Destructive/absent capabilities are simply NOT in the catalogue (e.g. delete-account does not exist), which is how AST-03 "it can't" is enforced.
- Provide a `@ngrx/store-devtools` registration (dev only) for inspectability.
</design_decisions>
